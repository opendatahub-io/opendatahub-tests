# TC-E2E-003: Job cancellation during execution — submit, admit, run, cancel

**Objective**: Verify the end-to-end flow of submitting a job, allowing it to be admitted and start running, then cancelling it while it is executing.

**Preconditions**:
- Kueue Operator installed on the cluster
- ClusterQueue `eval-cq` created with nominalQuota: cpu=1, memory=4Gi
- LocalQueue `eval-queue` created in test namespace mapped to `eval-cq`
- EvalHub deployed and accessible

**Test Steps**:
1. **Setup**: Create ClusterQueue (cpu=1, memory=4Gi) and LocalQueue
2. **Submit**: POST /api/v1/evaluations/jobs with queue specification
3. **Verify admission**: Wait for Workload `Admitted=True`, EvalHub reports `running`
4. **Cancel**: DELETE /api/v1/evaluations/jobs/{id}
5. **Verify cancellation response**: HTTP 204
6. **Verify Kubernetes Job**: Job is deleted or terminated
7. **Verify Workload cleanup**: Workload resource is removed or shows terminal state
8. **Verify quota released**: ClusterQueue's `flavorsUsage` decreases by the job's request
9. **Verify LocalQueue**: admittedWorkloads decremented
10. **Submit new job**: POST /api/v1/evaluations/jobs — verify it can be admitted (quota released)
11. **Teardown**: Delete all remaining resources, LocalQueue, and ClusterQueue

**Expected Results**:
- Job is successfully cancelled while running (HTTP 204)
- Kubernetes resources are cleaned up (Job terminated, Workload removed)
- Quota is fully released after cancellation
- A new job can be admitted using the freed quota
- The complete flow demonstrates: submit → admit → run → cancel → quota release → new job admitted
